# 재설계안: runAutoMatch를 반복 개선(local search) 구조로 전면 재작성

## 0. 이 문서에 대해

이 문서는 지금까지의 순차 패치 파이프라인(1~13단계, 재시도·구제·신규파티·재구제·병합·
aggressive 등)을 대체할 **대안 알고리즘**의 설계도입니다. "매칭엔진 모듈분리" 작업으로
`src/lib/matchEngine.js`가 별도 파일로 분리되고 있으므로, 이 재설계안은 **그 파일의
대안 버전**(`matchEngine.experimental.js` 등)으로 구현하고, 관리자 파일의 import 한 줄을
바꿔 기존 구현과 비교할 수 있게 합니다. 기존 `matchEngine.js`는 건드리지 않습니다.

`[Inference]` 표시는 확정 사실이 아닌 제안이며, 결과 기준(9절)만 지켜지면 구체적 구현
방식은 재량입니다. 이 문서의 서술은 설계 제안이며 실행 검증된 것이 아닙니다.

## 1. 기획서 근거 — 무엇이 고정이고 무엇이 자유인지

기획서 2.10절 "자동 매칭 실행 시 처리"는 다음 고수준 순서만 규정합니다:

1. 최신 신청 데이터 불러오기
2. **일반 신청 우선 배정**
3. **지원 신청 빈자리 배정**
4. **파티별 전투력 균형 조정**
5. 동일 대표 캐릭터 시간 중복 검사
6. 결과 생성 → 미배정 캐릭터 분류

**기획서는 이 6단계를 어떤 알고리즘으로 구현할지 규정하지 않습니다.** "균형 조정"이 별도
단계로 명시된 것 자체가, 초기 배치와 균형 개선을 분리해도 되고 반복적으로 수행해도 됨을
시사합니다. 즉 **재설계는 기획서 위반이 아니라, 기획서의 6단계를 하나의 정합적 엔진으로
구현하는 방식**입니다.

기획서가 고정한 것 (반드시 지켜야 함):

- 파티 인원: 탱커 1 + 서포터 1 + 나머지 딜러 (2.6절)
- 동일 대표 캐릭터는 같은 시간에 1명만 (1.9, 1.14, 2.17절 — 저장 차단 항목)
- 지원 신청은 서로 다른 시간에 반복 배정 가능, 배정 의무 없음 (1.9, 1.14절)
- 신청하지 않은 캐릭터·시간은 후보가 아님 (1.14절)
- 미배정 사유는 정해진 카테고리를 따름: "동일 대표 캐릭터가 해당 시간에 이미 배정됨",
  "일반 신청 캐릭터가 다른 시간에 이미 배정됨", "역할 자리 부족", "파티 정원 부족",
  "선택 시간에 적합한 파티 없음" (2.15절) — 이번 재설계도 사유 문구를 이 범주에 맞춘다.
- 일반 신청: 원문은 "최대 한 번만 배정"(1.9절)이지만, 이 프로젝트에서는 이후 "정확히
  1회 필수"로 강화 확정됨(10.1절, 2026-07-08) — 재설계는 강화된 버전을 따른다. 이는
  기획서의 상위 호환이며 위반이 아니다.

기획서가 규정하지 않아 이 프로젝트에서 자체적으로 정한 것 (재설계 시 유지할 확정 사항):

- 파티 수 = 딜러 기준 ceil, 지원은 파티 수 미증가
- **딜러 0 시간대는 파티를 생성하지 않는다 — 탱커 인원수와 무관** (수정안 5.1절: "딜러
  잉여가 존재할 때만" 신규 파티 생성. 2026-07-08 확정. 이전에 있었던 "탱커 2명 이상이면
  파티 생성" 규칙은 이때 폐기되었다 — 이 재설계안 초안에 실수로 되살아나 있었던 것을
  이번에 바로잡음)
- **서포터 초과분: 교차 배치하지 않는다** (딜러·탱커 슬롯 진입 불가), 항상 사유와 함께
  미배정 (수정안 4.1·10.2절, 2026-07-08 확정 — "혼합 풀에 초과서포터 포함" 규칙은 이때
  폐기되었다 — 마찬가지로 이 재설계안 초안의 오류였음)
- **탱커 초과분**: 빈 딜러 슬롯에만 배치 가능(실제 역할 표시 유지), 안 되면 미배정, 새
  파티 생성 트리거 아님 (수정안 4.2·10.3절)
- **딜러 초과분**: 기존 파티 빈 슬롯 우선 사용 → 그래도 남으면 신규 파티 생성 검토
  (지원 서포터 탐색 우선 → 없으면 대표 비충돌 시간에 딜러만으로 생성, 서포터 슬롯은
  빈자리) (수정안 5~8절·10.4절)
- 지원 채우기: 본래 역할 빈 슬롯 우선, 없을 때만 교차(서포터 슬롯은 지원 서포터만),
  강캐(전투력 높은) 우선 배정
- 딜러 집중 배치(1단계 딜러 시간 선택 유지). 탱커·서포터의 "분산" 조건(수정안 2절 3항)은
  동률 처리에만 적용한다는 우선순위 재확인 (2026-07-08)
- 서포터 시간확보 메커니즘(2026-07-10 확정): 서포터의 남은 후보 시간이 전부 무파티
  시간이면, 같은 대표 딜러를 재배치해 자리를 확보한다
- 패널티·저항압력 기반 최종 전투력 공식(`charFinalPower`)
- 일반+지원(both) 신청 처리 규칙 — 일반 배정이 최종 실패하면 지원 후보에서도 제외 (2026-07-08)

## 2. 문제를 다시 정의: 순차 파이프라인 → 단일 최적화 문제

### 2.1 현재 구조의 근본 문제 (재설계 동기)

현재는 "이 캐릭터는 몇 시"를 1단계에서 조기 확정하고, 그 결과가 파티 구성 관점에서
나쁘면(서포터가 무파티 시간에 갇힘, 딜러 1명짜리 잔여 파티, 빈 탱커에 다른 시간 딜러를
못 끌어옴 등) 이름 붙은 패치(재시도·구제·신규파티·재구제·병합·aggressive)를 계속
덧붙이는 구조다. 사각지대가 발견될 때마다 새 패치가 필요할 위험이 있다 `[Inference]`.

### 2.2 재정의

"어느 캐릭터가 몇 시·어느 파티·어느 슬롯에 들어가는가"를 **하나의 배치(assignment)로
놓고, 목적함수 점수를 반복적으로 개선하는 문제**로 다룬다. 시간 배정과 파티 구성을
분리된 순차 단계가 아니라 **같은 상태를 두고 반복적으로 조정 가능한 하나의 문제**로 합친다.

---

## 3. 상태 표현

```
assignment = {
  // 캐릭터별 현재 배치 (없으면 미배정)
  charPlacement: Map<charKey, { time, partyId, slotRole, slotType } | null>,
  // 시간·파티별 슬롯 구성 (파티 존재 여부 자체도 가변 상태)
  parties: Map<partyId, { time, slots: [{role, occupant|null}, ...] }>,
}
```

`[Inference]` 파티는 고정된 목록이 아니라 **상태의 일부**로 취급한다 — 파티 생성·삭제·
시간 이동도 이웃 탐색 연산의 결과로 일어날 수 있게 한다. 이렇게 하면 "신규 파티 생성"이
별도 특수 케이스가 아니라 일반적인 이동 연산의 자연스러운 결과가 된다.

## 4. 목적함수 (우선순위 계층 — lexicographic)

이번 대화에서 축적된 확정 사항을 종합하면 아래 우선순위가 자연스럽게 도출된다
`[Inference — 여러 개별 확정을 하나의 우선순위로 합성한 것]`:

1. **최우선 (사실상 하드 제약): 일반(및 both) 신청 캐릭터의 미배정 수 최소화.**
   미배정이 남더라도 "역할 자리 부족" 등 정당한 사유가 있어야 하며(3절 물리적 불가능
   케이스 제외), 배정 가능한데 안 하는 상태는 점수상 큰 페널티.
2. **2순위: 사용된 파티(딜러 슬롯 기준) 수의 "이론적 최소치 대비 초과분" 최소화.**
   (2026-07-10 정정 — 실측 보고서 "목적함수 파티수 초과분 재정의 요청" 확정) 처음에는
   단순 "사용된 파티 수" 자체를 2순위로 뒀으나, 이러면 파티 수를 1개 줄이기 위해 표준편차를
   몇 배씩 악화시키는 선택이 "더 나은 결과"로 채택되는 문제가 실측으로 확인됐다. 파티 수가
   이론적 최소치(딜러 기준 ceil)를 지키는 상태들끼리는 이 항목이 항상 0으로 동률 처리되고,
   그 이상으로 파티가 많을 때만 페널티를 받는다 — "불필요한 파티 생성 방지"라는 원래 취지를
   그대로 살리면서, 이미 최소치를 지킨 상태들 사이에서는 3순위(균형)가 자연스럽게 결정한다.
3. **3순위: 파티 간 최종 전투력 균형(표준편차 최소화).** 탱커·서포터·딜러 모두 대상,
   콘텐츠 전체 시간대 통합 (5.1절 확정 범위).
4. **4순위(별도 서브 문제): 지원 신청자 배치.** 1~3순위로 일반 배치가 수렴한 뒤, 남은
   빈자리를 지원으로 채우는 것은 별도 후속 최적화로 둔다 (지원은 배정 의무가 없고 파티
   수를 늘리지 못하므로, 일반 배치를 흔드는 변수로 섞지 않는 것이 안전하다 `[Inference]`).

동일 대표 동일 시간 제약, 신청 시간 범위 제약은 **탐색 자체에서 위반 상태를 생성하지
않는 방식(연산자 정의 시점에 차단)으로 처리**한다 — 목적함수의 페널티항으로 두면 위반
상태가 일시적으로 나타날 수 있어 권장하지 않는다 `[Inference]`.

## 5. 초기해 생성 (Construction)

빈 상태에서 시작해 반복 개선만으로 수렴시키는 것은 비효율적이므로, 현재 로직의 아이디어를
**초기해를 빠르게 만드는 용도로만** 재사용한다:

1. 딜러를 신청 시간 적은 순으로, 가능한 한 기존에 딜러가 배치된 시간에 우선 배치 (딜러
   집중 배치 아이디어 재사용)
2. 탱커·서포터를 신청 시간 적은 순으로, 딜러 있는 시간 우선 배치
3. 동일 대표 동일 시간 제약 위반 시 그 캐릭터는 이 단계에서 미배정으로 남김 (이후 반복
   개선이 처리)
4. 딜러 수 기준으로 파티 틀 생성 (파티 수 = ceil(딜러/슬롯), 딜러 0이면서 탱커 2명 미만인
   시간은 파티 생성 안 함)
5. 슬롯 배치: 서포터→탱커→딜러 순 (수정안 3절), 초과분은 4절의 교차 배정 규칙대로 혼합
   배치

이 초기해는 **지금의 1~3단계와 사실상 동일**하다 `[Inference]`. 차이는 이후 이걸 고정된
결과로 보지 않고, 다음 단계에서 계속 흔든다는 점이다.

## 6. 이웃 탐색 연산자 (Move Operators)

반복 개선 루프가 한 번에 시도할 수 있는 "상태를 한 단계 바꾸는" 동작들. 모든 연산은
적용 후 하드 제약(동일 대표 동일 시간, 신청 시간 범위, 파티 정원)을 위반하면 안 된다.

- **TimeMove**: 캐릭터 하나를 자신의 다른 신청 시간으로 옮긴다 (파티가 없으면 새로
  생성, 옮긴 뒤 원래 자리가 비면 그 파티가 완전히 비었을 때 파티 자체를 삭제). **단, 새
  파티 생성은 그 시간에 딜러가 최소 1명 있을 때만 유효하다** — 딜러 없는 시간에 탱커나
  서포터만으로 새 파티를 만드는 이동은 하드 제약 위반으로 간주해 애초에 후보에서 제외한다
  (1절의 "딜러 0 시간대 파티 미생성" 규칙을 연산자 정의 단계에서 강제하는 것 — 목적함수
  페널티가 아니라 연산 자체를 막는 방식이어야 한다는 4절 원칙과 일치).
- **Swap**: 같은 역할의 두 캐릭터가 서로의 자리(시간·파티)를 맞바꾼다.
- **RoleCrossMove**: 확정된 교차 배정 규칙 내에서, 캐릭터를 다른 역할 슬롯으로 옮긴다
  (딜러↔탱커, 서포터는 자기 슬롯만).
- **PartySplit/Merge**: 같은 시간대의 두 파티를 인원 이동으로 합치거나(현재의 "파티 병합"),
  한 파티를 둘로 나눈다 (거의 쓰이지 않겠지만 완전성을 위해 정의).

`[Inference]` 지금까지 나온 "재시도", "구제 재배치", "aggressive Pass 0/A/B", "동일 시간대
균형 재배치"는 전부 이 4개 연산자 중 하나(주로 TimeMove·Swap)의 특수 사례다. 통합하면
이름 붙은 여러 패스 대신 하나의 탐색 루프가 이들을 전부 포괄한다.

## 7. 반복 개선 루프

```
1. 초기해 생성 (5절)
2. 개선 없음 카운터 = 0
3. while 개선 없음 카운터 < 임계값 and 반복 횟수 < 상한:
     a. 현재 상태에서 적용 가능한 연산(6절)들을 후보로 나열
        (미배정 캐릭터 관련 연산 우선 탐색 — 4절 1순위 반영)
     b. 각 후보를 적용했을 때의 목적함수 점수(4절, lexicographic 비교) 계산
     c. 가장 개선되는 연산을 적용 (동률이면 무작위 선택 `[Inference]`)
     d. 개선이 없으면 개선 없음 카운터 증가, 있으면 0으로 리셋
4. 수렴하면 지원 채우기(4순위) 별도 실행 — 현재의 "지원 역할 우선 + 강캐우선" 로직 재사용
```

`[Inference]` 힐클라이밍(매번 개선되는 수만 선택)으로 시작하고, 지역 최적에 자주 갇히면
낮은 확률로 개선 안 되는 수도 받아들이는 방식(simulated annealing 유사)을 추가하는 것을
권장한다. 다만 이 프로젝트 규모(길드원 50명 미만)에서는 순수 힐클라이밍 + 여러 번 다른
초기해로 재시작(random restart 소수 회)만으로도 충분할 가능성이 높다 `[Inference — 이
프로젝트 규모를 감안한 추정이며 실측 필요]`. 계산 시간이 체감상 느리면(수 초 이상) 반복
상한이나 재시작 횟수를 줄이는 식으로 조정한다.

## 8. 지원 채우기 (4순위, 별도 후속 단계)

일반 배치가 수렴한 뒤:

- 서포터 슬롯은 지원 서포터만, 탱커·딜러 슬롯은 지원 캐릭터 역할 무관 교차 가능 (본래
  역할 우선, 없을 때만 교차 — 기존 확정)
- 전투력 높은 지원자부터, 미배정 지원자 전원 1회 기회 후 반복 허용 (기존 확정)
- 지원 채우기는 파티 수를 늘리지 않는다 (새 파티·새 시간대 생성 없음)

### 8.1 수정안 6~8절의 단순화 채택 (사용자 확정, 2026-07-10)

수정안 6~8절은 "딜러 잉여로 신규 파티를 만들 때, 지원 서포터를 구할 수 있는 시간을
먼저 찾고, 그 시간에 서포터까지 포함해 파티를 만들라"는 절차를 규정한다 (지원 서포터
유무에 따라 6절/7절 두 경로로 분기).

**확정: 단순화 버전을 채택한다.** 일반 배치 단계(1~3순위)는 지원 서포터 유무를 신경 쓰지
않고 그냥 1~3순위 기준으로 가장 좋은 시간에 신규 파티를 만들고(딜러만 채워진 파티로),
이후 8절 지원 채우기 단계가 그 파티의 빈 서포터 슬롯도 다른 빈 서포터 슬롯과 동일하게
취급해 채운다. 지원 서포터가 그 시간에 있으면 채워지고, 없으면 빈 슬롯으로 남는다 —
수정안 6절(지원서포터 있는 경우)과 7절(없는 경우)이 별도 분기가 아니라 **같은 절차의
자연스러운 결과**가 된다.

"두 시간 후보가 1~3순위 기준으로 동률이면 지원 서포터 있는 쪽을 선호"하는 수정안의
세부 선호는 반영하지 않는다 — 8절이 사후에 채우므로 실질적 결과 차이가 크지 않다고
판단해 구현 단순성을 우선한 것으로, 사용자가 직접 확정했다.

이는 수정안의 **절차**(6절/7절의 2단계 분기)를 그대로 옮기는 대신, 수정안이 지키려던
**규칙**(딜러 잉여만 신규 파티 트리거, 가능하면 지원 서포터로 채움)만 반영형 목적함수와
후속 단계 구조로 재현한 것이다 — 이 재설계안 전체의 원칙과 일치한다.

## 9. 완료 기준 — 지금까지 발견된 모든 사각지대를 패치 없이 통과해야 함

새 엔진은 아래 실제 사례들을 **개별 이름의 패치 코드 없이, 6~7절의 단일 루프만으로**
통과해야 한다:

1. **서포터 시간확보 사례** (히란야·임땡 그룹, 2026-07-10 회귀 테스트): 같은 대표 딜러들이
   시간을 다 차지해 서포터의 유일한 남은 시간이 무파티 시간이 되는 경우 — TimeMove로 딜러를
   재배치해 해소되어야 한다.
2. **딜러 1/2 잔여 파티** (22:30 사례): 딜러 3명을 2자리 파티에 나눌 때 1명이 남는 것은
   산술적으로 불가피하므로, 이 경우는 "실패"가 아니라 "이미 최선"임을 확인하는 회귀
   테스트로 포함한다 (더 줄이려 시도하면 안 됨).
3. **구제 후 placedNormal 미등록 크래시** — 새 구조에서는 `charPlacement` 단일 상태만
   있으므로 이런 이중 장부 불일치 자체가 구조적으로 발생하지 않아야 한다.
4. **신규 파티가 지원보다 늦게 생겨 일반이 밀리는 문제** — 지원 채우기가 일반 수렴 이후
   별도 단계(8절)이므로 구조적으로 발생하지 않아야 한다.
5. **딜러 0 시간대는 탱커 인원과 무관하게 파티가 생기지 않는다** — 탱커만 2명 이상 있고
   딜러가 0명인 시간대를 만드는 테스트 데이터로, 새 엔진이 그 시간에 파티를 만들지 않고
   해당 탱커·서포터를 사유와 함께 미배정 처리하는지 확인한다 (6절 TimeMove의 딜러 0 제약이
   실제로 지켜지는지 검증하는 항목).
6. 기존에 통과했던 모든 요청 프롬프트의 완료 기준(패널티 계산, 저항압력 공식, both 처리,
   신청 유형 3종, 교차 배정 규칙, 역할 우선 지원 채우기)도 동일하게 충족한다.
