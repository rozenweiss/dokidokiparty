# 클라이언트 48시간 자동삭제 루프 제거 요청 프롬프트 (중복 삭제 경로 해소)

> 이 문서는 분석 세션에서 확인한 문제와 확정 조치를 구현 세션에 전달하기 위한
> 것입니다. 대상 코드 위치·원문은 2026-07-08 업로드본
> `GuildPartyMatcherAdmin.jsx`(2518줄) 기준입니다. `[Unverified]` 작업 시점의
> 최신 소스와 행 번호가 다를 수 있으니 원문 대조 후 수정하세요.

## 1. 배경과 문제

48시간 만료 매칭 데이터 자동 삭제가 현재 **두 경로**로 공존한다:

1. **서버**: `Code.gs`의 `cleanExpiredResults()` — 시간 기반 트리거로 실행
   (별도 작업 "방법 A"로 추가됨).
2. **클라이언트**: `GuildPartyMatcherAdmin.jsx`의 `setInterval` 루프
   (약 2375~2389행) — 관리자 화면이 열려 있는 동안 60초마다 만료 콘텐츠를
   찾아 `purgeContentData()` 실행.

두 경로는 같은 48시간 기준을 쓰므로 **같은 만료 시점에 함께 발동하도록
상관되어 있다.** `[Inference]` 위험 시나리오: 서버 트리거는 실행 시작 시점에
읽어둔 행 번호로 `deleteRow`를 수행하는데, 그 읽기~삭제 사이에 클라이언트
루프가 같은 `results:*` 행을 먼저 지우면 행 번호가 밀려 **서버가 엉뚱한 행
(rep:* 또는 guild-config일 수 있음)을 삭제할 수 있다.** 발생 확률은 낮지만
kv가 source of truth이므로 결과가 심각한 유형이다.

또한 파일 상단 주석(약 16행)은 "48시간 자동 삭제 타이머는 구현하지 않고,
수동 삭제 버튼만 제공합니다"라고 되어 있어 실제 코드(루프 존재)와 모순이다 —
서버 트리거로 이전하면서 클라이언트 루프 제거가 누락된 상태로 판단된다.

## 2. 확정 조치

**클라이언트 자동삭제 루프를 제거하고, 자동 삭제는 서버 트리거
(`cleanExpiredResults`) 단일 경로로 일원화한다.** 수동 "매칭 삭제" 버튼과
`purgeContentData` 함수 자체는 그대로 유지한다 (수동 삭제·콘텐츠 삭제에서
계속 사용됨).

## 3. 수정 내용 (`GuildPartyMatcherAdmin.jsx` 단독)

### 3.1 자동삭제 `useEffect` 제거 (약 2375~2389행)

아래 블록 전체를 삭제한다:

```jsx
// 48시간 자동 삭제: 앱이 열려 있는 동안 주기적으로 만료 여부를 확인합니다.
useEffect(() => {
  const t = setInterval(async () => {
    const now = Date.now();
    const expired = config.contents.filter((c) => {
      const meta = resultsMeta[c.id];
      return meta && now - meta.generatedAt >= RETENTION_MS;
    });
    if (expired.length === 0) return;
    for (const c of expired) await purgeContentData(c);
    showToast(`자동 매칭 후 48시간이 지나 ${expired.map((c) => c.name).join(", ")}의 신청/매칭 데이터가 자동 삭제되었습니다.`);
    await refresh();
  }, 60000);
  return () => clearInterval(t);
}, [config.contents, resultsMeta, refresh, showToast]);
```

- 바로 위의 `useEffect(() => { refresh(); }, [refresh]);`는 남긴다.
- 삭제 후 이 컴포넌트에서 더 이상 쓰이지 않게 되는 것이 있는지 확인:
  `RETENTION_MS` 상수(약 249행)는 `[Unverified]` 다른 사용처가 있는지 확인 후,
  이 루프에서만 쓰였다면 함께 제거한다. (서버 쪽 `cleanExpiredResults`가
  자체적으로 48시간 상수를 가지므로 프론트 상수와의 동기화 의무는 없다.)
  `resultsMeta`는 화면 표시에도 쓰일 가능성이 높으므로 삭제하지 말 것.

### 3.2 상단 주석 정정 (약 16행)

```
- 48시간 자동 삭제 타이머는 구현하지 않고, 수동 삭제 버튼만 제공합니다.
```
을 실제 구조에 맞게 수정한다. 예:
```
- 48시간 자동 삭제는 프론트가 아니라 Apps Script 시간 트리거
  (cleanExpiredResults)가 담당합니다. 프론트에는 수동 삭제 버튼만 있습니다.
```

### 3.3 데이터관리 탭 안내 문구 (약 2263행) — 유지

"자동 매칭 실행 후 48시간이 지나면 자동으로 삭제됩니다" 문구는 서버 트리거가
동작하는 한 여전히 사실이므로 그대로 둔다.

## 4. 작업·배포 순서 (중요)

자동삭제 공백이 생기지 않도록 아래 순서를 지킬 것:

1. **먼저** Apps Script 편집기 → 트리거 탭에서 `cleanExpiredResults`가
   시간 기반 트리거로 **등록되어 있는지 확인**한다. 미등록이면 등록부터 한다
   (예: 1시간 간격). 현재 상태에서는 클라이언트 루프가 유일한 자동삭제
   경로일 수 있기 때문이다.
2. (권장) 트리거가 최소 1회 정상 실행됐는지 실행 기록에서 확인한다.
3. 그 다음 프론트에서 루프를 제거하고 배포한다.
4. `Code.gs`는 이번에 수정하지 않으므로 Apps Script 재배포는 불필요하다.

## 5. 완료 기준 (검증 체크리스트)

- [ ] 관리자 화면을 열어둔 상태에서 60초 주기의 만료 확인 동작(네트워크
      요청·토스트)이 더 이상 발생하지 않는다.
- [ ] 수동 "매칭 삭제" 버튼은 기존대로 동작한다 (setMany → delete →
      syncMirror 흐름 유지).
- [ ] 콘텐츠 삭제 시 `purgeContentData` 연쇄 정리가 기존대로 동작한다.
- [ ] Apps Script 트리거 탭에 `cleanExpiredResults`가 등록되어 있고 실행
      기록에 실패가 없다.
- [ ] `[Unverified]` 만료 검증: 테스트용으로 `generatedAt`을 48시간 이전으로
      조작한 `results:*` 항목이 트리거 실행 후 kv와 미러 탭에서 제거되고,
      해당 콘텐츠 신청이 rep 데이터에서 걸러진다 — 기대 동작이며 실제
      환경에서 확인 필요.
- [ ] 빌드 검증: esbuild로 JSX 문법 확인 (`loader: 'jsx', jsx: 'automatic'`).

## 6. 남는 한계 (기록)

`[Inference]` 이 조치는 "만료 자동삭제끼리의 충돌" 경로를 제거하는 것이며,
서버 트리거 실행 중에 관리자가 **수동으로** 같은 콘텐츠의 매칭 삭제를 누르는
경우의 겹침 가능성은 여전히 0이 아니다 (트리거 실행은 수 초 이내로 짧아
실질 위험은 낮다고 판단). 근본 해결은 LockService 도입이며, 이는 기존 결정대로
보류 상태를 유지한다.
