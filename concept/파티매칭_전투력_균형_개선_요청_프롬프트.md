# 요청: 길드 파티 자동 매칭 알고리즘 개선 (전투력 균형)

## 0. 이 문서에 대해

이 프롬프트는 다른 세션(다른 대화)에 그대로 붙여넣어 사용하기 위해 작성되었습니다.
아래 내용은 실제 프로젝트 파일(`GuildPartyMatcherAdmin.jsx`, 기획서)을 직접 읽고 옮긴 것이며,
지어내거나 추측으로 채운 부분은 `[Inference]`로 표시했습니다. `[Inference]`가 붙은 항목은
기획서에 명시적으로 적혀 있지 않고, 코드/기획서 정황으로부터 유추한 내용이므로 답변자가
확정하기 전에 사용자에게 재확인하는 것을 권장합니다.

## 1. 현재 알고리즘 (실제 코드, 수정 없이 그대로 인용)

파일: `GuildPartyMatcherAdmin.jsx`

```js
function timeSlots(start, end, interval) {
  const [sh, sm] = start.split(":").map(Number);
  const [eh, em] = end.split(":").map(Number);
  let cur = sh * 60 + sm;
  const endMin = eh * 60 + em;
  const out = [];
  while (cur <= endMin) {
    out.push(`${String(Math.floor(cur / 60) % 24).padStart(2, "0")}:${String(cur % 60).padStart(2, "0")}`);
    cur += interval;
  }
  return out;
}

function finalPower(basePower, pressure) {
  if (!pressure || pressure <= 0) return basePower;
  return Math.round(basePower * (1 + pressure / 1000));
}

/* 신청서로부터 (캐릭터×시간) 매칭 후보 목록 생성 */
function buildCandidates(content, reps) {
  const out = [];
  Object.entries(reps).forEach(([repName, data]) => {
    (data.applications || []).forEach((app) => {
      if (app.contentId !== content.id || app.status === "cancelled") return;
      (app.characterIds || []).forEach((cid) => {
        const char = (data.subs || []).find((s) => s.id === cid);
        if (!char || char.active === false) return;
        (app.times || []).forEach((t) => {
          out.push({ repName, char, type: app.type, time: t, appId: app.id });
        });
      });
    });
  });
  return out;
}

/* 단순 휴리스틱 자동 매칭 */
function runAutoMatch(content, reps) {
  const dealerSlots = Math.max(content.partySize - 2, 0);
  const slotOrder = ["tank", "support", ...Array(dealerSlots).fill("dealer")];
  const allTimes = timeSlots(content.startTime, content.endTime, content.interval);
  const candidates = buildCandidates(content, reps);
  const usedNormal = new Set();
  const parties = [];
  const unassigned = [];

  allTimes.forEach((t) => {
    const pool = candidates.filter((c) => c.time === t && !(c.type === "normal" && usedNormal.has(c.char.id)));
    if (pool.length === 0) return;
    const byRep = {};
    pool.forEach((c) => {
      const cur = byRep[c.repName];
      if (!cur) { byRep[c.repName] = c; return; }
      const score = (x) => (x.type === "normal" ? 1000000 : 0) + finalPower(x.char.power, content.pressure);
      if (score(c) > score(cur)) byRep[c.repName] = c;
    });
    const picked = Object.values(byRep);
    const byRole = (role) => picked.filter((c) => c.char.role === role).sort((a, b) => finalPower(b.char.power, content.pressure) - finalPower(a.char.power, content.pressure));
    const tanks = byRole("tank"), supports = byRole("support"), dealers = byRole("dealer");

    const partiesCount = Math.max(
      tanks.length, supports.length,
      dealerSlots > 0 ? Math.ceil(dealers.length / dealerSlots) : 0,
      (tanks.length + supports.length + dealers.length) > 0 ? 1 : 0
    );
    if (partiesCount === 0) return;

    const timeParties = Array.from({ length: partiesCount }, (_, i) => ({
      time: t, partyNumber: i + 1,
      slots: slotOrder.map((role) => ({ role, nickname: null, repName: null, characterId: null, type: null })),
    }));

    tanks.slice(0, partiesCount).forEach((c, i) => { timeParties[i].slots[0] = { role: "tank", nickname: c.char.nickname, repName: c.repName, characterId: c.char.id, type: c.type }; });
    tanks.slice(partiesCount).forEach((c) => unassigned.push({ ...c, reason: "역할 자리 부족" }));
    supports.slice(0, partiesCount).forEach((c, i) => { timeParties[i].slots[1] = { role: "support", nickname: c.char.nickname, repName: c.repName, characterId: c.char.id, type: c.type }; });
    supports.slice(partiesCount).forEach((c) => unassigned.push({ ...c, reason: "역할 자리 부족" }));

    // 딜러는 파티0→1→2→...→2→1→0 순서(스네이크 드래프트)로 한 자리씩 배분
    let di = 0;
    for (let s = 2; s < slotOrder.length && di < dealers.length; s++) {
      const forward = (s - 2) % 2 === 0;
      const order = forward ? [...Array(partiesCount).keys()] : [...Array(partiesCount).keys()].reverse();
      for (const p of order) {
        if (di >= dealers.length) break;
        const c = dealers[di++];
        timeParties[p].slots[s] = { role: "dealer", nickname: c.char.nickname, repName: c.repName, characterId: c.char.id, type: c.type };
      }
    }
    for (; di < dealers.length; di++) unassigned.push({ ...dealers[di], reason: "역할 자리 부족" });

    timeParties.forEach((tp) => {
      const missing = {};
      tp.slots.forEach((s) => {
        if (!s.nickname) missing[s.role] = (missing[s.role] || 0) + 1;
        else if (s.type === "normal") usedNormal.add(s.characterId);
      });
      const parts = [];
      if (missing.tank) parts.push(`탱커 ${missing.tank}명 부족`);
      if (missing.support) parts.push(`서포터 ${missing.support}명 부족`);
      if (missing.dealer) parts.push(`딜러 ${missing.dealer}명 부족`);
      tp.shortage = parts.length ? parts.join(" · ") : null;
      parties.push(tp);
    });
  });

  return { parties, unassigned, generatedAt: Date.now(), published: false };
}
```

## 2. 처리 순서 요약 (검증됨)

1. 콘텐츠의 시작~종료 시각 사이 시간대를 하나씩 순회 (시간대 간 서로 독립적으로 처리, 뒤 시간대가 앞 시간대 결과를 참고하지 않음)
2. 해당 시간대 신청 후보 중, 일반 신청으로 이미 다른 시간대에 배정된 캐릭터(`usedNormal`)는 제외
3. 같은 대표 캐릭터가 그 시간대에 캐릭터를 여러 개 신청했다면 1명만 남김 (우선순위: 일반 신청 > 최종 전투력)
4. 탱커/서포터/딜러로 나눠 전투력 내림차순 정렬
5. 필요 파티 수 = max(탱커 수, 서포터 수, ceil(딜러 수 / 딜러 자리 수), 인원 있으면 최소 1)
6. 탱커·서포터는 정렬 순서대로 파티 0,1,2...에 1명씩 배정, 넘치면 미배정("역할 자리 부족")
7. 딜러는 스네이크 드래프트(0→1→2→…→2→1→0)로 배분해 파티 간 전투력 편차를 줄이려 시도
8. 시간대가 끝나면 그 시간대에서 실제로 채워진 일반 신청 캐릭터만 `usedNormal`에 등록

## 3. 반드시 지켜야 하는 제약 조건 (기획서 원문 근거 포함)

이 제약은 기획서 1.9, 1.10, 1.14절에 명시되어 있으며 알고리즘을 바꾸더라도 위반하면 안 됩니다.

- **동일 대표 캐릭터는 같은 시간에 캐릭터 1명만 배정** (기획서 1.9 "동일 대표 캐릭터는 같은 시간에 한 캐릭터만 배정", 1.14 "중복 배정 안내")
- **일반 신청 캐릭터는 여러 시간을 신청했어도 최대 1번만 배정** (기획서 1.9 "일반 신청에서는 각 캐릭터가 최대 한 번만 배정")
- **지원 신청 캐릭터는 서로 다른 시간대에 반복 배정 가능** (기획서 1.9, 1.14)
- 파티 인원: 탱커 1명 + 서포터 1명 + 나머지 딜러 (기획서 2.6 "파티 인원" 항목)
- 마도 압력이 1 이상이면 `finalPower = round(basePower * (1 + pressure/1000))`로 보정된 값을 기준으로 계산 (기획서 2.6, 코드 `finalPower` 함수와 일치)
- 캐릭터는 자신이 실제로 신청한 (캐릭터, 시간) 조합의 범위 안에서만 배정 후보가 될 수 있음 — 신청하지 않은 시간에 배정하는 것은 금지 `[Inference: 기획서에 "선택하지 않은 캐릭터는 매칭 후보에 포함되지 않습니다"(1.14)만 있고, "신청 안 한 시간"에 대한 명시적 문장은 없지만 동일한 원칙이 시간에도 적용된다고 추론함]`

## 4. 현재 알고리즘의 한계 (개선이 필요한 지점)

- **시간대별 완전 독립 처리**: 각 시간대를 순서대로 하나씩 그리디(greedy)하게 처리하고 다른 시간대를 고려하지 않음. 어떤 시간대에서 먼저 소진된 자원(특히 대표 캐릭터의 대표-캐릭터 중복 제거 단계)이 이후 시간대의 배정 품질에 영향을 줄 수 있음.
- **대표 캐릭터 중복 제거가 "이 시간대 안에서" 즉시 확정됨**: 3단계(같은 대표 캐릭터 중복 제거)에서 "일반 신청 우선 → 전투력 높은 순"으로 그 시간대 안에서만 캐릭터를 고정함. 즉, 어떤 대표 캐릭터가 여러 캐릭터를 여러 시간에 신청했을 때, 그 대표 캐릭터가 "어느 시간에 어느 캐릭터를 낼지" 조합을 전체적으로 최적화하지 않고 시간대 단위로 근시안적으로 결정함.
- **파티 간 균형은 딜러 스네이크 드래프트에만 의존**: 탱커/서포터는 순서대로 그냥 채우기 때문에 탱커/서포터 전투력 편차는 전혀 고려되지 않음.
- **같은 시간대 안의 여러 파티 사이에서만 균형을 봄**: 다른 시간대의 파티들과는 비교하지 않음 `[Inference: 기획서에 "시간대별로 파티를 나열"한다는 표현(2.12)은 있지만, 시간대를 넘나드는 균형이 목표인지 여부는 명시돼 있지 않음 — 이 부분은 아래 5절의 질문으로 남겨둠]`

## 5. 개선 목표

**최종 목적**: 파티 간 최종 전투력이 편향되지 않고 골고루 분배되도록 자동 매칭 알고리즘을 개선.

**추가로 검토해달라고 요청하는 아이디어**: 한 대표 캐릭터가 여러 캐릭터를 보유하고 있고, 그 캐릭터들이 서로 다른 시간대에 걸쳐 신청되어 있는 경우, "이 대표 캐릭터가 어느 시간대에 어느 캐릭터로 참여할지"를 시간대별로 근시안적으로 고정하지 않고, 전체적인 전투력 균형을 더 좋게 만드는 조합으로 재배치(시간대 간 캐릭터 맞바꿈 포함)하는 것을 고려할 수 있는지 검토.

### 5.1 확정된 범위 (사용자 확인 완료 — 이 문서 작성자가 직접 답한 내용이며 추측이 아님)

- **균형을 맞추는 범위**: 같은 시간대 안의 파티들끼리만이 아니라, **콘텐츠 전체의 모든 시간대·모든 파티를 통틀어** 균형을 맞춘다. 즉, 20:00 파티들과 21:00 파티들 사이의 전투력 격차도 줄이는 것이 목표다.
- **캐릭터-시간 재배치 최적화 대상**: **일반 신청 + 지원 신청 캐릭터 모두** 대상으로 한다. (단, 3절의 제약 — 일반 신청은 최대 1회 배정, 지원 신청은 여러 시간에 반복 배정 가능 — 은 그대로 유지되어야 한다.)
- **균형 배정 대상 역할**: 딜러만이 아니라 **탱커·서포터·딜러 모두** 균형 배정의 대상으로 한다. (현재 코드는 탱커·서포터는 그냥 순서대로 채우고 딜러만 스네이크 드래프트를 적용하는데, 이 부분도 균형을 고려하도록 바꿔야 한다.)
- **운영 규모**: 길드원 50명 미만의 소규모 운영을 기준으로 한다. 계산량이 큰 최적화 기법(예: 정수계획법, 완전탐색)을 고려해도 되는 규모인지 이 규모를 기준으로 판단해달라고 요청한다.

예시 상황 `[Inference: 사용자 의도를 예시로 구체화한 것으로, 기획서에 있는 문장이 아님]`:
- 대표 캐릭터 "가나다"가 캐릭터 A(전투력 높음, 20:00/21:00 신청), 캐릭터 B(전투력 낮음, 20:00/21:00 신청)를 등록
- 현재 알고리즘은 20:00 시간대를 처리할 때 "전투력 높은 A"를 그 시간대의 대표로 고정해버림
- 만약 20:00 파티들은 이미 전반적으로 전투력이 높고 21:00 파티들은 낮다면, A를 21:00에 배정하고 B를 20:00에 배정하는 편이 전체 균형에 더 유리할 수 있음 — 하지만 현재 로직은 이런 시간대 간 트레이드오프를 전혀 보지 않음

## 6. 요청 사항

아래를 설계/제안해 주세요 (지금 이 세션에서는 코드 수정을 하지 않고, 알고리즘 설계와 근거만 정리해 주면 됩니다):

1. 위 제약 조건(3절)을 모두 지키면서, **콘텐츠 전체의 모든 시간대·모든 파티**를 통틀어 최종 전투력 편차를 줄이는 개선된 배정 알고리즘 설계안 (같은 시간대 내부만이 아니라 시간대 간 균형까지 다룰 것)
2. 대표 캐릭터의 캐릭터-시간 조합을 전역적으로 고려하는 방법 — **일반 신청과 지원 신청 캐릭터 모두**를 재배치 최적화 대상에 포함하되, 일반 신청은 최대 1회 배정 / 지원 신청은 여러 시간 반복 배정 가능이라는 제약(3절)은 유지 (예: 시간대를 순차 그리디로 처리하는 대신, 전체 후보를 한 번에 놓고 최적화하는 방식으로 바꿀 수 있는지, 혹은 그리디 결과에 대해 로컬 스왑/개선을 반복 적용하는 2단계 방식이 나은지 등 여러 접근을 비교)
3. **탱커·서포터·딜러 모두**에 대해 파티 간 균형을 고려하는 방법 (현재는 탱커·서포터는 균형 없이 순서대로만 채워지고 딜러만 스네이크 드래프트가 적용됨 — 이 비대칭을 어떻게 해소할지 포함)
4. "전투력 편차"를 측정할 구체적인 목적함수 제안 (예: 파티별 평균 전투력의 표준편차, 최대-최소 격차 등 후보를 비교하고 장단점 설명)
5. **길드원 50명 미만의 소규모 운영, 브라우저에서 즉시 실행**되는 환경을 기준으로, 정수계획법(ILP)이나 완전탐색 같은 무거운 최적화 기법까지 고려할 수 있는지, 아니면 휴리스틱/로컬서치 수준이 적절한지 판단
6. 알고리즘이 "최적해 보장"이 아니라 휴리스틱이라면, 그 한계를 명확히 설명 (이 프로젝트는 이미 그리디 휴리스틱을 사용 중이며, 완벽한 균형을 보장한다고 표현하지 않아야 함)

## 7. 범위 확정 완료

이전 버전에서 열려 있던 질문들은 사용자가 직접 답변하여 5.1절에 반영했습니다. 답변자는 아래 항목을 다시 사용자에게 묻지 않고 그대로 전제로 사용하면 됩니다.

| 질문 | 확정된 답변 |
|---|---|
| 균형 범위 | 콘텐츠 전체의 모든 시간대·파티를 통틀어서 |
| 재배치 최적화 대상 | 일반 + 지원 신청 모두 |
| 균형 배정 대상 역할 | 탱커·서포터·딜러 모두 |
| 운영 규모 | 소규모 (길드원 50명 미만) |

다만 이 표는 사용자가 이 대화에서 직접 선택한 답변을 그대로 옮긴 것이며, 그 이상으로 세부 사항(예: 정확한 시간대 수, 콘텐츠당 평균 신청자 수 등)은 확인된 바 없습니다. 설계 중 추가로 필요한 세부 수치가 있다면 추측하지 말고 사용자에게 다시 질문해 주세요.
