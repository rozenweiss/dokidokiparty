# 요청: 목적함수 2순위를 "파티 수"에서 "이론적 최소 초과분"으로 재정의 (옵션 A 확정)

## 0. 이 문서에 대해

구현 세션에 그대로 붙여넣어 사용하는 문서입니다. 베이스라인은 사용자가 제공하는 최신
`matchEngine_experimental.js`(균형최적화형)입니다. `[Inference]` 표시는 확정 사실이 아닌
추론이며, 구현 전 애매하면 사용자에게 확인해야 합니다. 이 문서의 서술은 코드 정독 기준이며
실행 검증된 것이 아닙니다.

배경 문서: `확인 요청: 균형최적화형 목적함수 "파티 수 vs 균형" 우선순위 재검토` (실측 보고서,
2026-07-10). 그 보고서의 옵션 A가 확정되었으며, 이 문서는 그 구현 사양이다.

범위: **`objectiveOf` 함수의 2번째 항목 정의만** 변경한다. 목적함수의 1번째(미배정 수)와
3번째(표준편차) 항목, 사전식 비교 방식(`better`), 이웃 탐색 연산자, 지원 채우기, 재시작
로직은 변경하지 않는다.

## 1. 확정된 문제 (실측 보고서 요약)

`20260709_test`(27계정) 실측 결과, 균형최적화형이 파티 수 1개를 줄이기 위해 표준편차를
2610.5 → 5291로(2배 이상) 악화시키는 경우가 관찰되었다. 원인은 `objectiveOf`가
`[미배정 수, used.length, stdev(avgs)]`를 그대로 사전식 비교하기 때문이다 — 파티 수가
1이라도 차이 나면 균형은 그 차이가 아무리 커도 무시된다.

## 2. 확정된 수정 (옵션 A, 사용자 확정 2026-07-10)

목적함수 2번째 항목을 **"사용된 파티 수"가 아니라 "이론적 최소 파티 수 대비 초과분"**으로
바꾼다:

```js
function objectiveOf(state) {
  let unassignedCount = 0;
  charInfo.forEach((info, key) => { if (!state.placement[key]) unassignedCount++; });
  const used = state.parties.filter((p) => p.slots.some((sl) => sl.charKey));
  const avgs = used.map((p) => { const { sum, count } = partyPower(p); return count ? sum / count : 0; });
  const minParties = theoreticalMinParties(state); // 아래 2.1절
  const excessParties = Math.max(0, used.length - minParties);
  return [unassignedCount, excessParties, stdev(avgs)];
}
```

이렇게 하면 "이론적 최소치를 지킨" 상태들끼리는 2번째 항목이 항상 0으로 동률이 되어,
3번째 항목(표준편차)이 자연스럽게 승부를 가른다. 최소치를 못 지킨 상태(불필요하게 파티가
많은 경우)만 여전히 우선 페널티를 받는다.

### 2.1 `theoreticalMinParties(state)` 정의

`[Inference]` 정확한 이론적 최소치 계산은 두 가지 수준으로 나뉜다 — 구현자가 선택하되,
결과 기준(4절)만 지키면 된다:

- **간단한 버전(전역 하한, 시간 제약 무시)**: 그 상태에서 배정된 딜러 총원 ÷ 파티당 딜러
  슬롯 수, 올림. `Math.ceil(totalAssignedDealers / dealerSlots)`. 시간·대표 제약을 무시한
  절대 하한이라 실제로는 달성 불가능할 수 있지만, "초과분"을 판단하는 기준으로는 충분히
  보수적이고 계산이 간단하다.
- **정밀한 버전(시간별 하한의 합)**: 현재 상태에서 각 시간대에 배정된 딜러 수를 세어
  `Math.ceil(그 시간 딜러 수 / dealerSlots)`를 시간별로 합산. 기존 확정 규칙("파티 수 =
  딜러 기준 ceil", 시간대별 산정)과 더 정합적이나 계산이 조금 더 필요하다.

권장은 **정밀한 버전**이다 — 기존 파티 수 확정 규칙과 개념적으로 일치하고, 딜러 집중
배치의 의도(불필요하게 여러 시간대로 딜러가 흩어지는 것 방지)와도 맞아떨어진다. 다만
간단한 버전으로 시작해 문제가 없으면 그대로 두어도 무방하다.

## 3. 영향 범위 확인

- `objectiveOf`는 `localSearch`(반복 개선 루프)의 `better()` 비교와, 재시작 간 최종 채택
  비교(`if (!finalState || better(o, finalObj))`) 양쪽에서 쓰인다 — 이번 수정으로 **양쪽
  모두** 자동으로 옵션 A 방식을 따르게 된다 (이게 옵션 C 대신 옵션 A를 택한 이유이기도
  하다 — 국소 수정이 아니라 근본 수정).
- `PartyMerge`(같은 시간대 파티 병합) 연산자는 실측 보고서에서 이미 정상 동작(균형 개선에
  기여)이 확인되었으므로 변경하지 않는다.

## 4. 완료 기준

- 이론적 최소 파티 수를 지키는 두 후보 상태 중, 표준편차가 더 낮은 쪽이 선택된다 (파티 수가
  같아도, 달라도 — 둘 다 최소치를 지키는 한 표준편차로 결정됨).
- 이론적 최소치보다 파티가 많은 상태는 여전히 페널티를 받아 우선순위에서 밀린다.
- `20260709_test` 재실행 시, 이전에 관찰됐던 "파티 16개·표준편차 5291" 같은 결과가 더 이상
  선택되지 않고, 표준편차가 낮은 "파티 17개" 계열의 결과가 채택된다 `[Inference — 기대
  결과이며 실행 검증 필요]`.
- 미배정 수 최소화(1번째 항목)는 여전히 절대 우선이다 — 이번 수정으로 미배정이 늘어나는
  일은 없어야 한다.
- 기존 지원 채우기(전역 균형·3회 제한), 재시작 다양성 개선, 이웃 탐색 연산자는 변경 없이
  그대로 동작한다.
