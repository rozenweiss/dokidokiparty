# 버그 수정 요청: ① 구제 재배치 후 스왑 단계 크래시 ② 신규 파티의 일반 우선 위반

## 0. 이 문서에 대해

구현 세션에 그대로 붙여넣어 사용하는 문서입니다. 베이스라인은 사용자가 제공하는 최신
`GuildPartyMatcherAdmin.jsx`(약 3,190줄, 파티 병합·aggressive 모드 포함 버전)입니다.
`[Inference]` 표시는 확정 사실이 아닌 추론입니다.

아래 두 버그는 **실제 운영 데이터(구글 시트의 27계정 신청 데이터, 콘텐츠
"타바르타스+에이렐+서큐")로 알고리즘을 직접 실행해 재현·확인**된 것입니다. 행 번호는
해당 버전 기준이며 최신 소스에서 재확인하세요.

범위: 아래 두 버그 수정만. 알고리즘의 다른 규칙(딜러 집중, 역할 우선, 파티 병합,
aggressive 패스 등)은 변경하지 않습니다.

## 1. [치명] 버그 ①: step1Failed 캐릭터가 구제되면 스왑 단계에서 크래시

### 증상
실데이터로 `runAutoMatch` 실행 시 균형 스왑 단계에서 `TypeError: Cannot read properties
of undefined (reading 'char')` 발생 (870행 부근 `info.char.role`). 매칭 전체가 실패한다.

### 원인 (코드 경로로 확인됨)
1. 1단계에서 모든 시간이 대표 충돌인 캐릭터는 `step1Failed`에만 보관되고
   `placedNormal`에는 등록되지 않는다.
2. 구제 재배치의 `placeRescued`는 `placedSlotOf[key]`를 기록한 뒤 `placedNormal`을
   갱신하는데, `const pnIdx = placedNormal.findIndex(...); if (pnIdx !== -1) ...` 구조라
   **step1Failed 출신은 갱신되지 않고 미등록 상태로 남는다.**
3. 스왑 단계가 `placedSlotOf`의 모든 키를 `placedNormal.find(...)`로 조회하면서
   `info`가 undefined인 채 `info.char.role`에 접근 → 크래시.

재현 데이터에서는 대표 Claude의 윙클(id48, step1Failed → 23:30 구제)이 이 경로를 밟는다.

### 수정
`placeRescued`에서 `pnIdx === -1`이면 해당 entry를 `placedNormal`에 push한다:
`{ repName, char, role: char.role, time: party.time, allowedTimes: entry.allowedTimes, type: entry.type }`
`[Inference]` 방어적으로 스왑 단계의 조회에도 `if (!info) return null;` + `.filter(Boolean)`
가드를 함께 두는 것을 권장하나, 근본 수정은 placedNormal 등록이다.

## 2. 버그 ②: 신규 파티가 구제 이후에 생겨 "일반 우선" 원칙 위반

### 증상 (실데이터 재현)
딜러 잉여 신규 파티 생성이 22:30에 파티를 만들었는데(딜러 Rubi + 지원 서포터 루현),
그 서포터 슬롯을 **지원 타입(이룸빛·지원)**이 채우고, 22:30을 신청한 **일반 서포터
잉땡·연궁은 미배정**으로 남는다. "일반 배정이 우선, 지원이 먼저 들어가면 안 된다"는
확정 원칙 위반이다.

### 원인
실행 순서가 `구제 재배치 → 딜러 잉여 신규 파티 생성 → 스왑 → 지원 채우기`라서,
구제가 끝난 뒤 생긴 신규 파티의 빈 슬롯을 미배정 일반 캐릭터가 재고려받지 못하고,
지원 채우기가 가져간다.

### 수정
**신규 파티 생성(및 파티 병합) 이후, 지원 채우기 전에** 미배정 일반(및 both) 캐릭터에
대한 구제/재시도 패스를 한 번 더 실행한다. `[Inference]` 구현 형태 제안: 현재 미배정
목록(및 아직 unassigned로 확정되지 않은 잔여 리스트)을 대상으로 기존 `tryRescue`를
재호출하는 2차 패스가 가장 작은 수정이다 — 이때 버그 ①의 수정(placedNormal 등록)이
선행되어야 안전하다. 세부는 구현자 재량이되, 결과 기준은 3절.

## 3. 완료 기준

- 실데이터(시트의 "타바르타스+에이렐+서큐" 신청)로 재매칭 실행 시 예외 없이 완료된다.
- 신규 파티의 빈 슬롯에 대해, 그 시간을 신청한 미배정 일반 캐릭터가 지원 타입보다
  먼저 배정된다. 재현 케이스 기준: 잉땡(21:10·22:30) 또는 연궁(22:30·22:10) 중 최소
  한 명 이상이 22:30 신규 파티의 서포터 슬롯에 일반으로 배정된다 `[Inference — 슬롯이
  1개뿐이면 둘 다는 불가할 수 있음. 남는 쪽은 사유와 함께 미배정 유지]`.
- 일반(및 both) 캐릭터가 조용히 사라지는 경로가 없다.
- 기존 동작(딜러 집중, 역할 우선 2단계, 강캐우선, 파티 병합, aggressive 패스, both 연동
  실패 규칙, charFinalPower)이 유지된다.

## 4. 참고 (이번 수정 범위 아님 — 기록용)

- 교차 배정 시 슬롯의 role이 캐릭터 실제 역할로 바뀌는 표시 방식 때문에, aggressive
  패스의 role 매칭이 원래 탱커 슬롯이었던 자리를 "탱커 슬롯"으로 인식하지 못하는 경우가
  있다 `[Inference — 재현 데이터의 고르코 케이스]`. 미배정 복구율에 영향이 있으므로
  추후 개선 후보로 남긴다.
- 위 수정 후에도 남는 미배정(재현 데이터 기준 1~3명)은 그리디 휴리스틱의 한계로, 전원
  배정이 가능한 전역 배치가 존재하더라도 찾지 못할 수 있다. 이는 버그가 아니라 알려진
  한계이며, aggressive 모드와 관리자 수동 조정이 담당한다.
